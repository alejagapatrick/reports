     374                 :            :  *
     375                 :            :  * Transactions are added when they are seen on the network (or created by the
     376                 :            :  * local node), but not all transactions seen are added to the pool. For
     377                 :            :  * example, the following new transactions will not be added to the mempool:
     378                 :            :  * - a transaction which doesn't meet the minimum fee requirements.
     379                 :            :  * - a new transaction that double-spends an input of a transaction already in
     380                 :            :  * the pool where the new transaction does not meet the Replace-By-Fee
     381                 :            :  * requirements as defined in BIP 125.
     382                 :            :  * - a non-standard transaction.
     383                 :            :  *
     384                 :            :  * CTxMemPool::mapTx, and CTxMemPoolEntry bookkeeping:
     385                 :            :  *
     386                 :            :  * mapTx is a boost::multi_index that sorts the mempool on 4 criteria:
     387                 :            :  * - transaction hash
     388                 :            :  * - descendant feerate [we use max(feerate of tx, feerate of tx with all descendants)]
     389                 :            :  * - time in mempool
     390                 :            :  * - ancestor feerate [we use min(feerate of tx, feerate of tx with all unconfirmed ancestors)]
     391                 :            :  *
     392                 :            :  * Note: the term "descendant" refers to in-mempool transactions that depend on
     393                 :            :  * this one, while "ancestor" refers to in-mempool transactions that a given
     394                 :            :  * transaction depends on.
     395                 :            :  *
     396                 :            :  * In order for the feerate sort to remain correct, we must update transactions
     397                 :            :  * in the mempool when new descendants arrive.  To facilitate this, we track
     398                 :            :  * the set of in-mempool direct parents and direct children in mapLinks.  Within
     399                 :            :  * each CTxMemPoolEntry, we track the size and fees of all descendants.

     402                 :            :  * children (because any such children would be an orphan).  So in
     403                 :            :  * addUnchecked(), we:

     406                 :            :  * - update all ancestors of the transaction to include the new tx's size/fee
     407                 :            :  *
     408                 :            :  * When a transaction is removed from the mempool, we must:
     409                 :            :  * - update all in-mempool parents to not track the tx in setMemPoolChildren
